<Callout variant="wip" title="Untested path">

This guide documents the intended architecture, but the end-to-end flow
(fork → swap registry → deploy) has not been verified yet. File paths and
commands are accurate against the current repo; the claim that everything
composes cleanly into a self-hosted site is not. Expect rough edges and
open an issue if something doesn't line up.

</Callout>

<DocSection title="Overview">

Ghost UI is a standalone Vite + React app that serves as both a reference
design language and an interactive component catalogue. You can fork it, swap
in your own registry and tokens, and deploy it as the documentation site for
your design language.

</DocSection>

<DocSection title="Clone the Repository">

```bash
git clone https://github.com/block/ghost.git
cd ghost
pnpm install
```

</DocSection>

<DocSection title="Swap Your Registry">

Ghost UI reads from `packages/ghost-ui/registry.json` — a shadcn-compatible
registry that defines all components, styles, and dependencies. Replace this
with your own:

```bash
# Replace the registry with your own
cp path/to/your/registry.json packages/ghost-ui/registry.json

# Or rebuild from your shadcn config
cd packages/ghost-ui
pnpm build:registry
```

The registry format follows the **shadcn registry specification** — each item
has a name, type, dependencies, and file contents. Ghost UI uses this to
render install commands, source previews, and dependency graphs.

</DocSection>

<DocSection title="Customise Theme Tokens">

Design tokens live in CSS custom properties. The main entry point is the
styles directory:

```text
packages/ghost-ui/src/styles/
├── base.css          # CSS variable definitions (light + dark)
├── globals.css       # Global resets and imports
└── ...
```

Edit `base.css` to change colours, spacing, border radii, typography scales,
and shadows. All components inherit from these tokens — changes propagate
everywhere automatically.

### Theme Presets

Ghost UI ships with theme presets in `src/lib/theme-presets.ts`. You can add
your own brand preset or modify existing ones. The live theme panel
(accessible via the UI) lets you experiment with token values in real time.

</DocSection>

<DocSection title="Update Component Demos">

Component demos live in `src/components/docs/primitives/` and
`src/components/docs/ai-elements/`. Each file exports a default component
that renders a demo for its corresponding UI component.

The component registry in `src/lib/component-registry.ts` maps component
slugs to their metadata (name, category, description, demo component). Add
or remove entries here to control what appears in the catalogue.

</DocSection>

<DocSection title="Fonts">

Ghost UI defaults to system fonts. To use a custom typeface:

1. Add your font files to the fonts directory
2. Update the `@font-face` declarations in `globals.css`
3. Update the `--font-display` and `--font-body` CSS variables

</DocSection>

<DocSection title="Development">

```bash
# Start the dev server
just dev
# or: cd packages/ghost-ui && pnpm dev

# The site is available at http://localhost:5173
```

</DocSection>

<DocSection title="Build & Deploy">

Ghost UI builds to a static SPA that can be deployed to any static hosting
provider:

```bash
# Build for production
just build-ui
# or: cd packages/ghost-ui && pnpm build

# Output is in packages/ghost-ui/dist/
```

The `dist/` directory contains a fully self-contained SPA. Deploy it to
Vercel, Netlify, Cloudflare Pages, GitHub Pages, or any static file server.

### Docker

For container deployments, serve the static output with any web server:

```dockerfile
FROM nginx:alpine
COPY packages/ghost-ui/dist/ /usr/share/nginx/html/
# Configure SPA fallback for client-side routing
RUN echo 'server { listen 80; location / { try_files $uri /index.html; } }' \
    > /etc/nginx/conf.d/default.conf
```

</DocSection>

<DocSection title="Connect to Ghost">

Once your design language is running as a Ghost UI site, you can use Ghost's
core tooling to profile the parent and track drift across consumers.
Profiling is a skill recipe your host agent runs; comparison and intent
tracking are deterministic CLI verbs.

1. Publish your `registry.json` at a stable URL.
2. Install the skill bundle into your host agent with `ghost emit skill`,
   then ask it to "profile this design language" pointed at your registry.
   The recipe writes `fingerprint.md` and validates it with `ghost lint`.
3. Have each consumer profile itself the same way, then run
   `ghost compare parent.fingerprint.md consumer.fingerprint.md` to
   measure drift. For PR-level gating, ask your host agent to "review
   this PR for design drift" — that's the `review` recipe, which you can
   pre-bake into a per-project slash command with `ghost emit review-command`.
4. Record intent with `ghost ack`, `ghost adopt`, or `ghost diverge` so
   drift with reasoning is preserved as signal, not noise.
